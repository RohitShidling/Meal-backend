================================================================================
EXECUTIVE SUMMARY
================================================================================
Critical correctness issues exist in payment finalization code (undefined
variables and an undefined helper), which would prevent reliable subscription
activation after successful gateway payment unless a different branch/version
is deployed. Separately, Firebase OTP integration uses a static reCAPTCHA
placeholder, Swagger may be exposed without the intended docs auth middleware,
and the admin SPA stores refresh tokens in localStorage (XSS / physical device
risk). Global API rate limits are permissive (1000 req/15m per IP on client
routes). Flutter caches cart and domain data in SharedPreferences (not
hardware-backed). Several cross-stack contract risks exist (subscription list
endpoint naming, cached prices vs server recomputation at cart add). A targeted
pass on cache and DB search found: in-memory PhonePe status cache remains
TTL-only without LRU (see A7); the unauthenticated HTML status-page discloses
payment outcome for anyone with the merchant transaction id (E4); sampled
admin/client search filters use parameterized ILIKE/LIKE (E5).

SECTION A — BACKEND (Node / Express / PostgreSQL)
================================================================================

A1. [P0 / CRITICAL] Broken payment subscription activation helpers
    Location: src/client/controllers/paymentController.js
    Evidence:
      - Line ~192-196: Query result stored as `existingSub`, then code uses
        `currentSubRes.rows[0]` — `currentSubRes` is never defined (runtime
        ReferenceError on successful payment finalization path).
      - Line ~245: `const nextSub = upsertRes.rows[0]` — `upsertRes` is never
        assigned from the INSERT ... RETURNING query.
      - Line ~247: `recordSubscriptionHistory(dbClient, ...)` — `dbClient` is
        undefined; `recordSubscriptionHistory` is not defined anywhere in repo
        (grep shows only this call site).
    Why it matters: Successful PhonePe webhook/redirect/status flows call
      finalizeSuccessfulPayment → activateSingleSubscription; this path
      appears non-functional in the audited source tree.
    Direction: Repair variable names, capture query result, implement or remove
      history helper, add integration tests against a test DB.

A2. [P1 / HIGH] Swagger UI mounted without docsAuthMiddleware
    Location: src/server.js (imports docsAuthMiddleware but route is)
      app.use('/api-docs', swaggerUi.serve, swaggerUi.setup(specs));
    docsAuthMiddleware exists at src/common/middlewares/docsAuthMiddleware.js
      but is NOT wired to /api-docs in the shown server.js snippet.
    Why it matters: When ENABLE_SWAGGER_DOCS=true, API surface may be browsable
      without Basic/Bearer protection (route inventory, request shapes).
    Direction: Wrap /api-docs with docsAuthMiddleware; disable Swagger in prod
      by default.

A3. [P1 / HIGH] Firebase phone auth — static reCAPTCHA token
    Location: src/common/services/otpService.js — sendOTP posts
      recaptchaToken: "server-side-bypass"
    Why it matters: Production Firebase projects typically require valid
      reCAPTCHA / App Check; bypass string may fail closed OR, if project is
      misconfigured, enable SMS OTP abuse and cost amplification.
    Direction: Align with Firebase recommended server flow; rate-limit per
      phone; monitor send volume.

A4. [P1 / HIGH] OTP session store is in-process only
    Location: src/common/services/otpService.js — sessionStore = new Map()
    Why it matters: Multi-instance deployments lose sessions; load balancer
      stickiness required; horizontal scale breaks verify; race under failover.
    Direction: Redis or signed opaque session ticket returned to client under
      strict TTL.

A5. [P2 / MEDIUM] Server startup logs env presence (information disclosure)
    Location: src/server.js — logs whether FIREBASE_API_KEY is loaded
    Why it matters: Confirms attack surface to log aggregators; minor.
    Direction: Log only in debug or use boolean without naming provider.

A6. [P2 / MEDIUM] Global client/common rate limit very high
    Location: src/server.js — limiter max: 1000 / 15 minutes for /api/client
      and /api/common
    Why it matters: Credential stuffing / enumeration still bounded by auth
      route sub-limiter (20/10m) but other endpoints (e.g. lookups) may be
      scraped aggressively.
    Direction: Tier limits per route class; stricter anonymous endpoints.

A7. [P2 / MEDIUM] Payment gateway status cache unbounded
    Location: src/client/controllers/paymentController.js — gatewayStatusCache
      = new Map() with TTL per key but no global max entries
    Why it matters: Long-lived process + many unique txn IDs → memory growth.
    Direction: LRU cap or periodic prune.

A8. [P2 / MEDIUM] Dynamic SQL fragment assembly (review-only)
    Location: src/common/services/mealEligibilityService.js — builds SQL with
      template literals for column aliases and predicates; mealController uses
      subscriptionActiveOnDatePredicateSql output in WHERE clause.
    Evidence suggests fragments are internal constants, not user input.
    Why it matters: If any future change pipes user input into those builders,
      injection risk appears quickly.
    Direction: Keep predicates in whitelisted constants; add static analysis

A9. [P2 / MEDIUM] resolveEntityName uses table name from fixed map

      `SELECT name FROM ${table} WHERE id=$1` with table from {child, teacher,

A11. [P3 / LOW] test_payment.js and scratch/check_admins.js

A13. [Integration] PhonePe webhook validation

SECTION B — ADMIN FRONTEND (React / Vite)
================================================================================

B1. [P1 / HIGH] JWT refresh + access tokens in localStorage
    Location: src/services/api.js — TokenService uses localStorage for
      admin_access_token, admin_refresh_token, admin_user JSON
    Why it matters: Any XSS in admin app → full account takeover; tokens survive
      tab close; not HttpOnly cookie model.
    Direction: Prefer secure HttpOnly SameSite cookies for refresh; short-lived
      access in memory; strict CSP.

B2. [P2 / MEDIUM] Long-lived persisted API cache in localStorage
    Location: src/services/api.js — PERSIST_CACHE_TTL_MS = 7 days for API JSON
    Why it matters: Stale master data (menus, prices) after admin edits; UX
      inconsistency with backend truth.
    Direction: Shorter TTL for mutable resources; ETag / cache-bust on mutate.

B3. [P2 / MEDIUM] Image data URLs cached in localStorage (size & privacy)
    Location: src/services/api.js — IMAGE_CACHE_PREFIX, blobToDataUrl path
    Why it matters: Quota exhaustion; shared machine leakage of menu imagery;
       not sensitive like PII but operational risk.

B4. [P3 / LOW] No dangerouslySetInnerHTML grep hits in admin src
    Observation: Reduces XSS surface from React escape; still validate any
      future rich-text fields.

B5. [Contract] BASE_URL from VITE_API_BASE_URL
    Location: src/services/api.js
    Why it matters: Misconfigured env points admin at wrong backend (similar to
      Flutter wrong IP) — operational not code flaw.

SECTION E — DATABASE & DATA PROCESSING (INFERRED FROM CODE)
================================================================================

E1. No standalone SQL migration directory in repo glob — schema evolution relies
    on initDB() idempotent DDL; risky for production roll-forward/rollback.

E2. Session timezone forced to Asia/Kolkata (default) in pool options — all
    DATE comparisons depend on this; document for operators; changing TZ
    without data migration affects eligibility.

E3. Heavy analytics controllers (e.g. subscriptionAnalyticsController.js) not
    deep-read in this pass — assumption: review for unbounded date range
    queries (needs runtime EXPLAIN).

E4. [P2 — Privacy / bearer capability] Unauthenticated payment status HTML page
    Location: src/client/controllers/paymentController.js — exports.statusPage
      GET /api/client/payment/status-page?tid=<merchantTransactionId>
    Behavior: No JWT; loads transaction by tid; calls same in-memory
      getCachedGatewayStatus(tid) as authenticated status checks; renders amount,
      entity display name, success/fail copy, and tid in HTML.
    Why it matters: Merchant transaction IDs act like opaque capability tokens.
      If leaked (referrer, screenshot, chat, guessable in a weak integration),
      a third party learns payment outcome and beneficiary label without auth.
      Page can also drive finalize paths when gateway reports terminal state while
      DB row is still pending (same class of trust as redirect callback).
    Direction: Require short-lived signed state from checkout, or session cookie,
      or redirect only to frontend with tid in fragment; rate-limit by IP on this
      route; avoid embedding sensitive labels without auth.

E5. [P2 / hygiene — DB search] Sampled list/search paths use bound parameters
    Evidence (grep/read): admin getAllPayments (search → ILIKE $n with value in
      params array); admin getSubscribedUsersRemainingMeals (q → ILIKE $n);
      admin/client school list (search → LIKE with $1/$2); subscription analytics
      filters append ILIKE $paramCount with pushed values.
    Why it matters: Reduces SQL injection risk vs string-concatenated predicates.
    Caveat: Not every dynamic WHERE in the repo was audited; future edits should
      keep all user-derived fragments as parameters or strict allowlists.

E6. [P3 — Consistency] In-memory PhonePe status cache vs authoritative DB
    Cross-ref: A7 (unbounded Map) — additionally, default TTL
      PAYMENT_STATUS_CACHE_TTL_MS (10s) can serve a stale gateway payload briefly
      after another code path already updated the row (mostly UX / sync timing).
    Direction: Invalidate cache entry on successful processSuccessfulTransaction
      / processFailedTransaction, or key store by (txnId, row version).

G1. [P2 / MEDIUM — Admin UI / XSS] Toast uses innerHTML with interpolated message
    Location: meal-admin-fronent/src/components/Toast.js
    Evidence: toast.innerHTML = `...${message}</span>` (message not escaped).
    Why: If any caller passes API error text or user-controlled strings containing
    HTML/script, DOM XSS is possible in the admin browser session.
    Direction: Use textContent, or React portal toast, or DOMPurify for HTML.

--------------------------------------------------------------------------------
G2. [P2 / MEDIUM — API contract / dead feature] "Admin" query on client teacher
    and professional profile routes is unreachable
    Location: src/client/middlewares/authMiddleware.js rejects any JWT where
    decoded.role !== 'client'.
    Location: src/client/controllers/teacherController.js and
    professionalController.js branches on req.user.role === 'admin' &&
    req.query.clientId.
    Evidence: Client tokens always carry role 'client'; admin tokens use
    ADMIN_JWT_SECRET and fail client middleware verification.
    Why: Swagger/docs imply admins can pass clientId on /api/client/teacher/*
    and /api/client/professional/* — that path does not work with normal auth.
    Direction: Either mount separate admin routes with adminAuthMiddleware, or
    remove dead branches and fix documentation.

--------------------------------------------------------------------------------
G3. [P3 / LOW — Flutter / analyzer] Widespread deprecated Color.withOpacity
    Evidence: withOpacity appears across many screens (home_screen, weekly_menu,
    profile screens, cart, etc.); analyzer flags deprecated_member_use in several
    files.
    Why: Future SDK breakage / inconsistent theming; not a runtime bug today.
    Direction: Migrate to withValues(alpha: ...) per Flutter guidance.

--------------------------------------------------------------------------------
G4. [P3 / LOW — Flutter] use_build_context_synchronously on subscription screen
    Location: lib/features/home/ui/screens/subscription_screen.dart — async
    _handleConnectivityChange uses context.read after await without a mounted
    check on context between await boundaries (analyzer info at ~line 75).
    Why: Rare race if widget disposed during fetch; can throw or mis-fire.
    Direction: if (!mounted) return after await; or use context.mounted.

--------------------------------------------------------------------------------
G5. [P3 / LOW — Backend / info disclosure] Firebase errors forwarded to client
    Location: src/client/controllers/authController.js — login/register send OTP
    catch blocks pass firebaseMsg into AppError(400), which is returned to
    clients for operational errors.
    Why: Can expose internal Firebase error codes/messages useful for probing.
    Direction: Map to generic user message; log detail server-side only.

--------------------------------------------------------------------------------
G6. [P3 / LOW — Backend] Profile controllers wrap unknown errors as AppError 500
    with error.message
    Location: teacherController, professionalController, parentController
    catch → next(new AppError(error.message || '...', 500)).
    Note: Global handler maps operational 500 responses to generic text in
    production — verify behavior for non-operational throws.
    Direction: Ensure only operational AppError used; avoid leaking driver text.

G7. [P3 — UX / a11y] Not systematically audited

G8. [P3 — UI consistency] Entity ID parsing in meal skip sheet
